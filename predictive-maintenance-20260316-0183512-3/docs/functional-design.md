# Functional Design: Preventive Maintenance with Inspection-Driven Corrective Work Orders

## 1. Overview

This document describes the functional design for configuring IBM Maximo 8.x to support inspection-based preventive maintenance where failed inspection points automatically create corrective work orders. The solution covers PM setup, inspection execution, automation rules, workflows, and user interaction.

## 2. Core Functional Components

1. **Assets and Locations**
   - Assets categorized into classes (e.g., PUMPS, MOTORS, COMPRESSORS).
   - Locations (areas, systems, lines) defined with proper hierarchy.
   - Asset templates and classification in place for consistent data.

2. **PM Records**
   - Calendar-based and/or meter-based PMs.
   - PMs associated to:
     - Asset (ASSETNUM) or
     - Location (LOCATION).
   - PMs tagged with:
     - Work Type: PM
     - Priority: default (e.g., 3)
     - PM status lifecycle (DRAFT, ACTIVE, INACTIVE).

3. **Inspection Forms / Task Lists**
   Two patterns are possible; the design can support either or both:
   - **Pattern A: PM Task List**
     - Use Job Plans associated with PMs.
     - Job Plan tasks represent inspection steps.
     - Additional attributes or custom fields capture PASS/FAIL.
   - **Pattern B: Maximo Inspections (recommended if licensed)**
     - Create Inspection Forms with questions and response options.
     - Associate Inspection Form to the PM.
     - Use Maximo Inspections application to capture results.

   For this solution, we assume **Maximo Inspections** is available.

4. **Work Orders**
   - PM WOs generated by PM engine (PMWOGEN or Cron Task).
   - Corrective WOs generated automatically as follow-up WOs:
     - WOTYPE = CM.
     - Relationship to PM WO via ORIGRECORDID and ORIGRECORDCLASS or custom relationship.
   - New fields (if needed):
     - `INSPECTIONREF` (reference to inspection result or question ID).
     - `SEVERITYLEVEL` (derived from inspection response).

5. **Automation Rules**
   - Automation script on inspection result saving or WO status change:
     - Evaluate inspection readings and responses.
     - Create one or more corrective WOs if failure criteria are met.
   - Validation automation script on PM WO:
     - Prevent status change to COMP if mandatory inspections are incomplete.
     - Ensure that critical failures are addressed.

6. **Workflow Integration**
   - Work Order Workflow managing WO status transitions (WAPPR → APPR → INPRG → COMP).
   - Decision nodes where:
     - If inspection indicates failure and corrective WO not created, route to Supervisor.
     - Supervisor can approve, request more info, or close with justification.

## 3. User Roles and Use Cases

### 3.1 Roles

- **Inspector / Technician**
  - Access PM WO.
  - Perform inspections and record results.
  - May see prompts or system messages suggesting corrective WO creation.

- **Planner**
  - Configure PMs, Job Plans, and Inspection Forms.
  - Review generated corrective WOs, adjust planning/assignments.

- **Supervisor**
  - Approve high-priority corrective WOs.
  - Review exceptions when failures have no corrective WOs.

- **Reliability Engineer**
  - Analyze inspection failure data.
  - Adjust PM frequencies and inspection questions.

### 3.2 Primary Use Cases

#### UC1: Execute Preventive Maintenance with Inspection

1. PM generates a PM WO for an asset.
2. Technician opens PM WO (via desktop or mobile).
3. Technician performs inspection guided by Inspection Form.
4. Technician records:
   - Pass/Fail responses,
   - Numeric readings,
   - Comments and attachments (photos).
5. Technician sets WO status to COMP when done, subject to validation rules.

#### UC2: Automatic Corrective WO Creation from Inspection Failure

1. Technician records a “Fail” on a critical question (e.g., “Leak Detected?” = YES).
2. Upon saving results, automation script evaluates the response.
3. System automatically creates a new Corrective WO:
   - Linked to same asset/location.
   - Work Type = CM.
   - Priority determined by severity (e.g., leak size or risk rating).
   - Short Description derived from question and response.
   - Long Description includes reference to original PM WO and inspection question.
4. Technician can see the follow-up WO in the related records section.
5. Planner/Supervisor reviews the corrective WO and schedules execution.

#### UC3: Validation Before Completing PM WO

1. Technician attempts to change PM WO status to COMP.
2. Automation script checks:
   - Whether all mandatory inspection questions are answered.
   - Whether any critical failures exist without a corrective WO.
3. If checks fail, system:
   - Prevents status change.
   - Displays message listing missing entries or unaddressed failures.
4. Technician corrects data or notifies Supervisor for exceptions.

#### UC4: Supervisor Exception Handling

1. System identifies that a critical failure has been recorded without corrective WO.
2. PM WO routed to Supervisor via workflow.
3. Supervisor reviews:
   - Inspection results.
   - Risk and operational context.
4. Supervisor can:
   - Approve auto-generated corrective WO (if not yet approved).
   - Create manual corrective WO and link it.
   - Override and allow closure with justification (e.g., asset decommissioned soon).

## 4. Functional Rules and Logic

### 4.1 Failure Criteria

Examples (configurable):

- Any question of type YES/NO with label “Safety Critical” and response “NO” or “FAIL” triggers a corrective WO.
- Numeric reading above/below configured thresholds (e.g., vibration > 10 mm/s).
- Qualitative rating “Poor” on defined questions.

Configuration elements:
- A lookup table or domain mapping inspection question IDs/labels to:
  - Failure thresholds.
  - Severity levels.
  - Default work order classification and failure codes.

### 4.2 Priority and Severity Mapping

Example mapping:
- Severity 1 (Critical) → WO Priority 1.
- Severity 2 (High) → WO Priority 2.
- Severity 3 (Medium) → WO Priority 3.
- Severity 4 (Low) → WO Priority 4–5.

Severity is derived from:
- Question classification (Critical/High/Medium/Low).
- Response value.
- Asset criticality (e.g., from asset record).

### 4.3 Linking Rules

- Corrective WO fields:
  - `ORIGRECORDCLASS` = WORKORDER.
  - `ORIGRECORDID` = PM WOID.
  - `PARENT` can be set to PM WO to show a clear hierarchy.
- Custom field `INSPECTIONREF` (or use long description) to store:
  - Inspection Form name and question ID.
  - Reading that triggered the failure.

### 4.4 Notifications

Optional email or in-app notifications:
- When critical corrective WO is created.
- When PM WO is blocked from completion due to missing inspection data.
- Supervisor alerts for exception approvals.

## 5. Screens and UX Impacts

### 5.1 Work Order Tracking

- Additional tab or section “Inspection Results”.
- Related Records section listing generated corrective WOs.
- Status field with workflow-driven values (WAPPR, APPR, INPRG, COMP, CAN).

### 5.2 Inspections Application

- Inspectors see assigned inspections tied to PM WOs.
- Use of pick lists for consistent responses.
- Visual indicators of failed questions.

### 5.3 Mobile UX (if used)

- Offline-capable inspection forms.
- Quick toggles for Pass/Fail with ability to attach photos.
- Automatic prompts when a “Fail” is selected for critical questions.

## 6. Reporting and KPIs

Key reports and dashboards:

1. **PM Compliance Report**
   - PMs due vs. completed.
   - PMs completed with full inspections vs. incomplete.

2. **Corrective WO from Inspections**
   - Count and percentage of PM WOs that generated corrective WOs.
   - Breakdown by asset class, location, and severity.

3. **Top Failure Modes**
   - Failure codes aggregated by asset/location.
   - Trend of specific inspection failures.

4. **Open Corrective WO by Origin PM**
   - For each PM WO, number of open corrective WOs.
   - Aged backlog by severity.

## 7. Security and Access Control (Functional)

- **Roles & Groups**
  - INSPECTOR: Create/update inspection results, view PM WOs, cannot modify PM configuration.
  - PLANNER: Maintain PMs, Job Plans, Inspection Forms; view all WOs.
  - SUPERVISOR: Approve WOs, manage workflow exceptions.
  - RELIABILITY: Read-only access to all inspection data, reporting tools.

- **Data Restrictions**
  - Limit inspectors to assets/locations they are responsible for.
  - Restrict changing of generated corrective WO critical fields (e.g., classification) to Planners/Supervisors.

## 8. Non-Functional Requirements (Functional Perspective)

- System should support:
  - Large volumes of inspection data without performance degradation.
  - Mobile access where available.
  - Audit logging on inspection result changes (who, when, before/after).

## 9. Dependencies

- Maximo 8.x core Work Management and PM modules.
- Maximo Inspections module (or equivalent).
- Enterprise integration (ERP, notifications, reporting) as required.
